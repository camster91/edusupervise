# docker/.env — DO NOT COMMIT (already in .gitignore at repo root)

This file exists for `docker compose` interpolation only.

## Why it's here

`docker-compose.yml` uses `${EDUSUPERVISE_RUNTIME_PASSWORD}`,
`${EDUSUPERVISE_SYSTEM_PASSWORD}`, and `${POSTGRES_PASSWORD}` inside
`environment:` blocks to build per-service `DATABASE_URL` strings
(runtime role for web, system role for worker + cron, owner role for
postgres initdb).

Docker compose interpolates `${VAR}` from the **HOST environment at the
moment `docker compose up` is invoked**. The `env_file:` directive loads
values INTO the container, but it is processed AFTER the `environment:`
block — so env_file does NOT contribute to interpolation. The host shell
on the VPS does not have these vars set, so the URLs were being built
with empty passwords, and the /api/health endpoint reported
`db: down` even though the web container was running.

This file lives in `docker/` next to `docker-compose.yml` and is
auto-loaded by `docker compose` for interpolation. It is gitignored.

## The canonical secrets live in `/root/edusupervise-secrets/.env`

That file is loaded into the container via the `env_file:` directive.
It is the source of truth for secrets on the VPS. Keep the values in
this `docker/.env` in sync with that file.

When adding a new interpolated variable to `docker-compose.yml`:

1. Add the value to `/root/edusupervise-secrets/.env` (canonical).
2. Add the value to `docker/.env` (compose-time interpolation).
3. Restart the affected service: `docker compose -f
   docker/docker-compose.yml up -d --force-recreate <service>`.

## Deploy command (VPS)

```bash
cd /opt/edusupervise
docker compose -f docker/docker-compose.yml pull --ignore-pull-failures  # optional
docker compose -f docker/docker-compose.yml up -d --force-recreate web
# or for the full stack:
docker compose -f docker/docker-compose.yml up -d --force-recreate
```

No need to source the env file or pass `--env-file` — `docker/.env` is
auto-loaded.

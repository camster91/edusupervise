# Phase 2 — PDF Schedule Ingestion

**Date:** 2026-07-04
**Status:** Spec ready, ready for swarm dispatch
**Owner:** Coder agent (assigned per swarm dispatch)
**Depends on:** Phase 1 (solo onboarding wizard creates the duty table to ingest into)
**Blocks:** Phase 3 (school-wide adoption is faster if admin can ingest PDF in bulk)

## Goal

A teacher uploads a PDF of their school's 5-day duty rotation (the kind Jason pulls from the board website) and the system extracts duty name, time, and location into a review card. The teacher confirms, edits if needed, and the rows are committed. Time-to-first-duty drops from ~10 minutes (manual form) to ~30 seconds (drop, confirm, done).

## Why pdfplumber, not vision models

Jason's PDFs (and 80% of district-published duty schedules) are structured tables. pdfplumber's `extract_tables()` handles them in <500ms without any AI, costs nothing per request, and doesn't hallucinate. Reserve vision models (Claude / GPT-4V) for the 20% of boards that publish image-only PDFs — and only build that fallback AFTER we see users with image-only PDFs.

## What Phase 2 needs to add

### 2.1 PDF upload endpoint

**Route:** `apps/web/app/routes/api.onboarding.upload-pdf.ts` (POST, CSRF-protected, rate-limited 10/hour/user)

**Behavior:**
- Accept multipart/form-data with field `file` (PDF only, max 10MB).
- Reject non-PDF (magic-byte check, not just MIME type — defense-in-depth).
- Store the file to `/data/uploads/{school_id}/{user_id}/{uuid}.pdf` with the SHA-256 manifest.
- Hand off to the parser in 2.2.
- Return JSON: `{ jobId, status: 'parsing' }` — async via the worker queue OR sync if parsing is <1s.

**File changes:**
- `apps/web/app/routes/api.onboarding.upload-pdf.ts` — new action.
- `apps/web/server/uploads.server.ts` (new) — helper to stage the file and compute SHA-256.
- Reuse existing rate-limit middleware (`server/rate-limit.server.ts`).

### 2.2 pdfplumber table extraction

**Library:** `pdfplumber` (Python) — needs to be available in the web container. Add to `docker/Dockerfile.web`:
```dockerfile
RUN pip install pdfplumber==0.11.*
```

**Server module:** `apps/web/server/pdf-parser.server.ts` (new)

**Behavior:**
- Open the staged PDF, iterate pages.
- For each page, call `page.extract_tables()`. Score each table by:
  - Number of cells with teacher-name-like patterns (Capitalized Lastname, optionally comma + Firstname)
  - Column header pattern matching ("Day 1", "Day 2", "Mon", "Tue", "Period 1", "Block A", "Recess")
- Pick the highest-scoring table.
- Return rows as: `[{ cycleDay, teacherName, location?, time? }]`
- For Jason's format (Day 1-4 columns), each row maps to one cycle day.
- For the time/location tables in his second image (Early Entry 8:45-9:00 at Kiss N Ride), detect the time pattern and emit a `recurringDuty` row instead.

**File changes:**
- `apps/web/server/pdf-parser.server.ts` — new, ~150 LOC.
- `docker/Dockerfile.web` — add pdfplumber install.

### 2.3 Review card UI

**Route:** `apps/web/app/routes/onboarding.pdf-review._index.tsx` (new)

**Behavior:**
- After upload, redirect to `/onboarding/pdf-review?jobId=...`.
- Loader fetches the parsed rows from the parser's output (stored in Redis or a `pdf_jobs` table — recommend Redis with 24h TTL).
- Render a table:
  ```
  | Day 1          | Day 2        | Day 3        | Day 4        |
  | Attwood        | McVey        | Attwood      | Papaikono    |
  | EA             | EA           | EA           | EA           |
  | Cyriac, L, S   | --           | --           | --           |
  | ...            | ...          | ...          | ...          |
  ```
- Each cell is editable — inline `<input>` or `<select>` for role (Teacher / EA).
- Bottom: "Save and continue" button → POSTs to `/api/onboarding/confirm-pdf` which writes all the rows.
- "Cancel" returns to the wizard.

**File changes:**
- `apps/web/app/routes/onboarding.pdf-review._index.tsx` — new route + UI.
- `apps/web/app/routes/api.onboarding.confirm-pdf.ts` — new action.

### 2.4 Concurrency model

**Choice:** Inline parsing for v1 (pdfplumber is fast), async worker for v2 (only if real PDFs take >2s).

For v1:
- The upload endpoint awaits the parser, stores the result in Redis under `pdf:{jobId}` with 24h TTL, returns `{ jobId, status: 'ready', rowCount }`.
- The review page reads from Redis.
- Confirm writes duties + assignments in one transaction.

For v2 (only if needed):
- Move parser to a worker queue (BullMQ on the existing Redis).
- Upload returns `{ jobId, status: 'parsing' }` immediately.
- Frontend polls or uses SSE for ready status.

**Don't build v2 unless v1's p95 parse time exceeds 2 seconds.**

### 2.5 Out-of-scope for Phase 2

- Vision-model fallback for image-only PDFs — Phase 2.5 if there's signal.
- Bulk import for admins (Phase 3).
- Calendar invites (Phase 4+).

## Risks and mitigations

| Risk | Mitigation |
|---|---|
| District PDF formats vary wildly | Start with the top 5 Canadian boards (Toronto, York, Peel, Ottawa-Carleton, Vancouver). Test against real samples from each. |
| pdfplumber fails on multi-column layouts | Provide a manual-edit fallback on every cell of the review card. |
| Table extraction picks the wrong table | Score by header pattern + cell-name density. Reject scores below threshold; ask user to re-upload. |
| PDF contains scanned image, not text | Detect with `pdfplumber.is_empty` (no text layer). Surface "this PDF is scanned; please upload a text version" until v2 lands. |
